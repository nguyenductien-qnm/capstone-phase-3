Báo cáo Nghiệm thu Mandate 5 - Runtime Hardening

Dự án: TechX Corp Microservices
Báo cáo viên: Châu Thành Trung (CDO-05)
Trạng thái: Hoàn thành

1. Tổng quan yêu cầu Mandate 5

Mandate 5 (Runtime Hardening) yêu cầu nhóm Task Force siết chặt an toàn bảo mật ở môi trường chạy (runtime) của các container trên cụm Kubernetes EKS, đồng thời thiết lập cơ chế chặn tự động ở cổng API Server (Policy-as-Code). Cụ thể bao gồm 4 yêu cầu lõi:

  • Cấm chạy quyền Root: Bắt buộc tất cả container khai báo runAsNonRoot: true, cấm leo thang đặc quyền allowPrivilegeEscalation: false, và loại bỏ Linux capabilities thừa (capabilities.drop: ["ALL"]).
  • Cấm tag trôi nổi (latest): Cấm dùng tag latest, dev hoặc không khai báo tag. Buộc sử dụng tag phiên bản cố định hoặc digest.
  • Khai báo tài nguyên bắt buộc: Bắt buộc khai báo đầy đủ cả 4 trường CPU & Memory requests/limits để tối ưu bin-packing cho cụm.
  • Bảo vệ tự động (Admission Controller): Tự động chặn và từ chối (Reject) ngay tại bước apply các manifest cấu hình vi phạm chính sách bảo mật.

2. Các công việc đã thực hiện

Chúng ta đã cấu hình thành công các giải pháp kỹ thuật trên mã nguồn cục bộ và chuẩn bị sẵn sàng môi trường nghiệm thu cho Mentor:

2.1. Cấu hình bảo mật Helm values.yaml

Đã cấu hình trực tiếp vào file values.yaml để áp dụng tự động các luật bảo mật:

  • Bảo mật mặc định (Global Default): Khai báo mặc định securityContext và podSecurityContext toàn cụm để 18+ services tự động kế thừa.
  • Sửa đổi cục bộ (Local Overrides): Bổ sung thủ công drop capabilities và cấm privilege escalation cho các service chạy User ID riêng biệt như frontend, frontend-proxy, payment, quote, kafka, valkey-cart.
  • Bảo vệ cho InitContainers: Thiết lập securityContext non-root (UID 1000) và đổi toàn bộ tag busybox sang tag cố định busybox:1.36.1 hoặc busybox:1.38.0.
  • Resource Sizing tối ưu (M5-07): Phân bổ đủ 4 trường CPU/RAM requests & limits cho toàn bộ các service với hạn mức tối giản (requests CPU 10m, RAM 32Mi cho backend) để duy trì ngân sách dưới $300/tuần.
  • ReadOnly Filesystem: Bật chế độ khóa ổ đĩa chỉ đọc readOnlyRootFilesystem: true riêng cho service ad để chứng minh khả năng siết bảo mật sâu theo dịch vụ.

2.2. Tài liệu Quyết định Kiến trúc & Đăng ký Ngoại lệ

Tạo các tài liệu thiết kế kiến trúc chuẩn hóa:

  • adr-admission-policy.md: Quyết định chọn OPA Gatekeeper làm Policy Engine chính thức.
  • adr-runtime-hardening.md: Thiết lập kế hoạch rollout an toàn (chạy Audit trước, Enforce sau) và đăng ký ngoại lệ có thời hạn cụ thể (hạn chót 31/12/2026) cho các core agents như otel-collector-agent và grafana sidecars.

3. Chi tiết cấu hình mã nguồn & Ý nghĩa từng dòng

Các chỉnh sửa chính được thực hiện tập trung tại file cấu hình: platform/charts/application/values.yaml. Dưới đây là chi tiết mã nguồn và ý nghĩa kỹ thuật của từng đoạn:

3.1. Bảo mật mặc định toàn cục (Global Default)

Đặt tại phần cấu hình chung toàn cụm ở dòng 37-48 để tất cả 18+ microservices tự động kế thừa:

platform/charts/application/values.yaml [Dòng 37-48]
securityContext:
  allowPrivilegeEscalation: false # Chặn tiến trình con tự nâng quyền lên Root
  runAsNonRoot: true # Bắt buộc container chạy bằng User thường, cấm chạy Root
  capabilities:
    drop:
      - ALL # Loại bỏ toàn bộ các quyền quản trị hệ thống Linux mặc định
podSecurityContext:
  runAsNonRoot: true # Áp dụng kiểm tra non-root cho toàn bộ Pod
  seccompProfile:
    type: RuntimeDefault # Sử dụng bộ lọc lệnh hệ thống an toàn mặc định của Docker/K8s
  • allowPrivilegeEscalation: false: Ngăn chặn các cơ chế nâng quyền đặc biệt (như lệnh sudo hoặc file SUID) giúp hacker không thể cướp quyền Root từ một user thường.
  • runAsNonRoot: true: Lệnh cho Kubernetes từ chối khởi động Pod nếu phát hiện container đòi chạy dưới quyền Root (User ID = 0).
  • capabilities.drop: [ALL]: Vứt bỏ các đặc quyền Linux nhỏ (như đổi giờ hệ thống, mở port hệ thống...) mà container không cần dùng tới, thu hẹp diện tấn công.
  • seccompProfile.type: RuntimeDefault: Kích hoạt bộ lọc syscall an toàn của hệ thống để chặn các cuộc gọi lệnh trực tiếp nguy hiểm lên hạt nhân máy chủ.

3.2. Ghi đè bảo mật cục bộ (Ví dụ: Frontend)

Sử dụng khi service có User ID đặc thù (như Next.js chạy User 1001) tại dòng 633-640:

platform/charts/application/values.yaml [Dòng 633-640]
    securityContext:
      runAsUser: 1001 # Chạy bằng User ID 1001 (Next.js user)
      runAsGroup: 1001 # Nhóm User ID 1001
      runAsNonRoot: true
      allowPrivilegeEscalation: false
      capabilities:
        drop:
          - ALL
  • runAsUser: 1001 & runAsGroup: 1001: Khai báo tường minh ID người dùng phi-root để Container chạy mã nguồn một cách hợp chuẩn.

3.3. Bảo vệ cho các InitContainers

Siết bảo mật và ghim tag cho các container chạy khởi tạo ban đầu (ví dụ: dòng 220-230):

platform/charts/application/values.yaml [Dòng 220-230]
    initContainers:
      - name: wait-for-kafka
        image: busybox:1.38.0 # Ghim phiên bản cố định (Cấm tag :latest trôi nổi)
        command: ["sh", "-c", "..."]
        securityContext:
          runAsUser: 1000 # Chạy init-container bằng User ID thường 1000
          runAsNonRoot: true
          allowPrivilegeEscalation: false
          capabilities:
            drop:
              - ALL

3.4. Định nghĩa Tài nguyên CPU & Memory (Resource Sizing)

Bắt buộc khai báo đủ 4 trường CPU/RAM requests & limits để vượt qua Gatekeeper tại dòng 213-219:

platform/charts/application/values.yaml [Dòng 213-219]
    resources:
      requests:
        cpu: 10m # Mức CPU tối thiểu yêu cầu (10 millicpu = 0.01 core)
        memory: 32Mi # RAM cam kết tối thiểu 32 Megabytes
      limits:
        cpu: 100m # CPU tối đa cho phép dùng (100 millicpu = 0.1 core)
        memory: 120Mi # RAM tối đa cho phép dùng (120 Megabytes)
  • requests: Dung lượng tài nguyên tối thiểu Kubernetes bắt buộc phải cấp cho Pod để khởi động. Đặt nhỏ giúp tiết kiệm chi phí chạy cụm (dưới $300/tuần).
  • limits: Dung lượng tối đa container được dùng. Nếu ram dùng lố quá limits, K8s sẽ tự động khởi động lại container (OOMKilled) để tránh kéo sập máy chủ.

3.5. Khóa ổ đĩa chỉ đọc (ReadOnly Root Filesystem)

Khóa cứng ổ đĩa của service ad để ngăn hacker ghi file độc hại tại dòng 258-265:

platform/charts/application/values.yaml [Dòng 258-265]
    securityContext:
      runAsUser: 1000
      runAsNonRoot: true
      allowPrivilegeEscalation: false
      capabilities:
        drop:
          - ALL
      readOnlyRootFilesystem: true # Khóa ổ đĩa ở chế độ Chỉ đọc

4. Bộ Test case Nghiệm thu của Mentor

Bộ test case được tạo sẵn dưới thư mục gatekeeper/tests/ để Mentor thực thi và tận mắt chứng kiến cơ chế chặn tự động hoạt động:

Tên File Test Loại Test Nội dung Vi phạm chính sách Trạng thái kỳ vọng
neg-01-root.yaml Chặn chạy root Không khai báo runAsNonRoot: true Bị Chặn (Reject)
neg-02-image-latest.yaml Chặn tag trôi nổi Sử dụng tag nginx:latest Bị Chặn (Reject)
neg-03-missing-resources.yaml Bắt buộc CPU/RAM Không khai báo resources.limits/requests Bị Chặn (Reject)
pos-01-valid.yaml Kiểm thử hợp lệ Cấu hình chuẩn an toàn và đầy đủ limits/requests Cho Phép (Pass)

5. Hướng dẫn Mentor kiểm thử thực tế

Lưu ý: Đảm bảo cụm EKS đã apply các Constraint và ConstraintTemplate trong thư mục gatekeeper/ trước khi chạy các lệnh kiểm thử sau.

Chạy các lệnh kubectl sau từ terminal máy khách:

# 1. Tạo namespace test cô lập
kubectl apply -f gatekeeper/tests/namespace-policy-test.yaml

# 2. Test kịch bản vi phạm quyền root -> Kỳ vọng bị BLOCK
kubectl apply -f gatekeeper/tests/neg-01-root.yaml

# 3. Test kịch bản vi phạm tag image latest -> Kỳ vọng bị BLOCK
kubectl apply -f gatekeeper/tests/neg-02-image-latest.yaml

# 4. Test kịch bản thiếu limits/requests -> Kỳ vọng bị BLOCK
kubectl apply -f gatekeeper/tests/neg-03-missing-resources.yaml

# 5. Test kịch bản hợp lệ -> Kỳ vọng tạo Pod THÀNH CÔNG
kubectl apply -f gatekeeper/tests/pos-01-valid.yaml
